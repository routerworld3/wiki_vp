Yes. In Juniper SSR/SVR, packet encryption is **not IPsec ESP** and **not TLS**.

It is best described as:

```text
SVR data-plane signaling + AES payload encryption + HMAC integrity
```

Juniper SSR encrypts **the session payload between SSR routers**, based on the configured security policy for that service/path. Juniper’s docs state that SSR provides policy-driven payload and metadata encryption between SSR routers, with encryption decisions defined per service and path. ([Juniper Networks][1])

---

## 1. What protocol is used?

The protocol is **SVR — Secure Vector Routing**.

But SVR is not a standard encryption protocol like:

```text
IPsec ESP
TLS
DTLS
MACsec
GRE over IPsec
```

Instead, SVR uses normal IP transport between SSR waypoint addresses and adds **SVR metadata** to packets for session signaling. Juniper describes SVR as using session detection, waypoint IPs, and metadata; the first packet carries metadata, and later packets follow the established session path without traditional tunnel overhead. ([Juniper Networks][2])

At a high level:

```text
Original packet enters SSR
   ↓
SSR classifies session
   ↓
SSR rewrites packet toward remote SSR waypoint
   ↓
SVR metadata is added to establish the session
   ↓
Payload is encrypted if policy requires it
   ↓
Remote SSR decrypts/restores/forwards native packet
```

---

## 2. What encryption algorithm is used?

The current SVR draft describes payload encryption as:

```text
AES-CBC-128
or
AES-CBC-256
```

The draft says plaintext is padded to a 16-octet boundary, which matches AES block size behavior. ([IETF Datatracker][3])

Juniper’s Session Smart security documentation says SSR uses AES encryption ciphers and a key plus initialization vector, with a unique IV generated per packet. ([Juniper Networks][1])

So practically:

```text
Encryption protocol/model: SVR payload encryption
Cipher: AES-CBC-128 or AES-CBC-256
Integrity/authentication: HMAC, commonly SHA-1, SHA-256, or SHA-256-128 depending on policy/config
Transport: normal IP/TCP/UDP between SSR waypoint addresses
```

---

## 3. Is it IPsec?

No.

With IPsec, the packet would usually be protected by ESP:

```text
Outer IP header
ESP header
Encrypted original IP packet / payload
ESP trailer / authentication
```

With SVR, the SSR does not build a traditional IPsec tunnel. It rewrites the packet toward SSR **waypoint addresses**, adds SVR metadata for session setup, and encrypts the **payload** when required.

Conceptually:

```text
Native packet:
10.10.10.10:50000 → 10.20.20.20:5000
Payload = Radio data

SVR packet between SSRs:
SSR-A waypoint → SSR-B waypoint
SVR metadata = original src/dst, service, tenant, policy, key info
Payload = encrypted radio data, if encryption policy requires
HMAC = integrity/authentication check
```

---

## 4. How is the encryption key handled?

At session start, the originating SSR generates a **payload key** for that session. The current SVR draft says a unique payload key is used per session/direction, generated by the originating router and carried in the encrypted portion of SVR metadata. It also says long-lived sessions may be rekeyed by sending new metadata mid-flow. ([IETF Datatracker][4])

Simple version:

```text
1. First packet hits SSR-A.
2. SSR-A decides this service requires encryption.
3. SSR-A generates a session payload key.
4. SSR-A encrypts the app payload.
5. SSR-A places the payload key inside encrypted SVR metadata.
6. SSR-B receives metadata, extracts key, stores it in session state.
7. SSR-B decrypts payload and forwards the original packet.
8. Future packets use the stored session key.
9. Long sessions can be rekeyed.
```

The draft also states that BFD is used for peer pathway reachability, quality measurement, peer authentication, and key management between SVR routers. ([IETF Datatracker][4])

---

## 5. What does HMAC do?

Encryption protects confidentiality.

HMAC protects integrity and peer trust.

Juniper’s docs say HMAC signatures are appended to the end of the packet payload to make sure the packet contents have not been changed and that the peering router is known and trusted. ([Juniper Networks][1])

So the security stack is roughly:

```text
AES = hide the payload
HMAC = detect tampering / authenticate packet integrity
SVR metadata = tell the remote SSR what session/policy/original addresses this packet belongs to
Waypoint rewrite = move traffic across the WAN without classic tunnel encapsulation
```

## 6. Packet-level view

### Before SSR-A

```text
Src IP:     10.10.10.10
Src Port:   50000
Dst IP:     10.20.20.20
Dst Port:   5000
Protocol:   UDP
Payload:    Radio-over-IP data
```

### SSR-A processing

```text
Classify:       Radio-over-IP service
Policy:         Allowed
Encryption:     Required
Cipher:         AES-CBC-256, for example
HMAC:           SHA256-128, for example
Path:           SSR-A waypoint → SSR-B waypoint
```

### Across WAN

```text
Outer visible packet:
Src IP:     SSR-A waypoint
Dst IP:     SSR-B waypoint
Protocol:   TCP or UDP depending on SVR/session behavior

SVR metadata:
Original source/destination
Tenant/service/policy
Security ID
Payload key information
Path/session information

Encrypted payload:
AES-encrypted radio data

HMAC:
Integrity/authentication value
```

### After SSR-B decrypts/restores

```text
Src IP:     10.10.10.10
Src Port:   50000
Dst IP:     10.20.20.20
Dst Port:   5000
Protocol:   UDP
Payload:    Original radio data
```

---

## 7. Does it encrypt every packet?

If the service policy says encryption is required, then yes, the **payload packets for that session** are encrypted between SSR endpoints.

The first packet also carries the SVR metadata needed to establish state. Later packets do not need full first-packet metadata, but they still follow the established session state and can continue using payload encryption. The SVR draft’s example states “all payload packets encrypted” after the key exchange/session setup. ([IETF Datatracker][4])

---

## 8. What if traffic is already TLS?

SSR can use **adaptive encryption**. In simple terms:

```text
If traffic already appears encrypted, such as HTTPS/TLS,
SSR may avoid encrypting it again.

If traffic is not encrypted,
SSR can encrypt the payload between SSR routers.
```

That is useful because it avoids wasting CPU/bandwidth on double encryption for HTTPS traffic, while still protecting legacy/non-TLS traffic like Radio-over-IP.

---

## 9. Bottom line

For Juniper SSR/SVR:

```text
Protocol name:
Secure Vector Routing, SVR

Is it IPsec?
No

Is it TLS?
No

Is it standard/open like IPsec?
No, it is vendor-specific/proprietary, though publicly documented in an IETF draft.

What encrypts the packet?
SVR payload encryption

Which cipher?
AES-CBC-128 or AES-CBC-256, depending on policy/config

What provides integrity?
HMAC, such as SHA1, SHA256, or SHA256-128 depending on policy/config

What carries key/session info?
SVR metadata in first-packet and, if needed, mid-flow metadata

What is visible on the WAN?
SSR waypoint-to-waypoint traffic, not the original end-device IP flow in the same native form
```

Simple mental model:

```text
IPsec encrypts packets inside a tunnel.
TLS encrypts an application session.
SVR encrypts selected session payloads between SSR routers using SVR metadata, AES, and HMAC.
```
