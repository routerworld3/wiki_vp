The one thing your original article downplayed

Service and Deployment are not connected at all. The Service does not look at the Deployment, does not know it exists, and does not care. The Service finds Pods by matching labels — the same Pods that happen to be created by the Deployment.

This means the chain is not Deployment → Service → Pod. It is:

Deployment ──labels──▶ Pod ◀──labels── Service

Two independent selector lookups meeting at the Pod. If either label match breaks, traffic silently drops. The LearnKube guide opens with this point — it is the most common source of confusion and the root cause of most "my service has no endpoints" tickets.

Match #1 — Deployment selector.matchLabels ↔ Pod template labels

Inside a single Deployment file, the spec.selector.matchLabels block tells the Deployment which Pods it owns. The spec.template.metadata.labels block stamps that label onto every Pod the Deployment creates. If these two blocks don't match exactly, Kubernetes rejects the Deployment.

A selector.matchLabels in Deployment spec ═══ template.metadata.labels on Pod
Deployment — spec.selector
Deployment — spec.template (Pod recipe)
# What the Deployment looks for apiVersion: apps/v1 kind: Deployment metadata: name: webapp spec: replicas: 2 selector: matchLabels: app: webapp # ← A template: ...
# What stamp goes on every Pod template: metadata: labels: app: webapp # ← A spec: containers: - name: webapp image: nginx:latest ports: - containerPort: 80
Why this match is enforced at apply-time: Kubernetes validates this when you run kubectl apply. A mismatch returns an error immediately — you cannot create a broken Deployment of this kind. This is the only one of the five matches that fails loud.

Match #2 — Service selector ↔ Pod labels

This is the critical match that connects external traffic to your Pods. Kubernetes does not validate this at apply-time. If the selector matches no Pods, the Service simply has zero endpoints and silently drops every request — no error, no warning, no log line.

B spec.selector in Service ═══ metadata.labels on Pod (from Deployment template)
Deployment — Pod template labels
Service — spec.selector
apiVersion: apps/v1 kind: Deployment metadata: name: webapp spec: selector: matchLabels: app: webapp template: metadata: labels: app: webapp # ← B spec: containers: - name: webapp image: nginx:latest
apiVersion: v1 kind: Service metadata: name: webapp-service spec: type: ClusterIP selector: app: webapp # ← B ports: - port: 80 targetPort: 80
The silent failure mode: If you change app: webapp to app: web-app on either side, your cluster reports everything as healthy. Pods are running. Service exists. NLB is up. But every request returns a TCP reset or a connection refused — because the Service's endpoint list is empty.

Verify match #2 with one command

kubectl describe service webapp-service | grep Endpoints
# Endpoints:   10.244.1.5:80, 10.244.2.7:80    ← good
# Endpoints:   <none>                          ← match is broken

Match #3 — Service targetPort ↔ Pod containerPort

The Service port is what clients connect to. The Service targetPort is where the Service forwards that traffic on the Pod. The targetPort value must equal the port your application actually listens on inside the container.

The containerPort field in the Pod spec is informational documentation only — but its value is what your app binds to, so it should be the same number as targetPort to avoid confusion.

C spec.ports[].targetPort in Service ═══ containerPort on Pod (the port the app listens on)
Deployment — container port
Service — port and targetPort
apiVersion: apps/v1 kind: Deployment metadata: name: webapp spec: template: spec: containers: - name: webapp image: nginx:latest ports: - containerPort: 80 # ← C # nginx listens on 80
apiVersion: v1 kind: Service metadata: name: webapp-service spec: selector: app: webapp ports: - port: 80 # external port — any number targetPort: 80 # ← C — must match app
Port vs. targetPort — the most confusing pair:
  • port: 80 — what clients use. Any number you like. Two Services can share the same port because they get different IPs.
  • targetPort: 80 — where traffic lands on the Pod. Must equal the app's listening port.

Many examples use port: 80 / targetPort: 8080 to make the distinction obvious. The values do not need to be equal — only targetPort needs to equal what the container app binds to.

Match #4 — Ingress backend.service.name ↔ Service metadata.name

The Ingress (or Istio VirtualService) references the Service by string name. A typo in this name produces a backend with no endpoints — the Ingress controller cannot find anything to route to.

D metadata.name on Service ═══ backend.service.name in Ingress
Service — metadata.name
Ingress — backend.service.name
apiVersion: v1 kind: Service metadata: name: webapp-service # ← D spec: selector: app: webapp ports: - port: 80 targetPort: 80
apiVersion: networking.k8s.io/v1 kind: Ingress metadata: name: webapp-ingress spec: rules: - http: paths: - path: / pathType: Prefix backend: service: name: webapp-service # ← D port: number: 80

Match #5 — Ingress backend.service.port.number ↔ Service port

The Ingress also names the Service port it wants to talk to. This must match the Service's external port field — not the targetPort.

E spec.ports[].port on Service ═══ backend.service.port.number in Ingress
Service — spec.ports.port
Ingress — backend.service.port.number
apiVersion: v1 kind: Service metadata: name: webapp-service spec: selector: app: webapp ports: - port: 80 # ← E (external) targetPort: 80
apiVersion: networking.k8s.io/v1 kind: Ingress metadata: name: webapp-ingress spec: rules: - http: paths: - path: / backend: service: name: webapp-service port: number: 80 # ← E pathType: Prefix
Easy mistake: If your Service has port: 80, targetPort: 8080 and your Ingress says port.number: 8080 — the Ingress fails. The Ingress speaks to the Service on the external port, not on the targetPort. The Service handles the port-to-targetPort translation internally.

What goes wrong when each match breaks

This is the lookup table for "my app deploys cleanly but nothing works." Read the symptom column, find your case, fix the match.

Match If broken, the symptom is... Diagnostic command What good output looks like Fix
A kubectl apply returns an error. Cannot create Deployment. kubectl apply -f deployment.yaml No error. Deployment created. Make selector.matchLabels identical to template.metadata.labels.
B Everything looks healthy. Requests time out or get connection refused. Silent failure. kubectl describe svc <name> | grep Endpoints Endpoints: 10.244.1.5:80, 10.244.2.7:80 Make Service selector identical to Pod's labels (from Deployment template).
C Endpoints exist but requests get connection refused from inside the cluster. kubectl exec <pod> -- netstat -ltn then check Service: kubectl get svc <name> -o yaml The port the app is listening on equals targetPort in Service. Set targetPort to the port your app actually binds to inside the container.
D Ingress shows backend with <error: endpoints not found>. kubectl describe ingress <name> my-service:80 (10.244.1.5:80, 10.244.2.7:80) Fix the Service name typo in Ingress backend.service.name.
E Ingress route 404s or 503s. Service direct port-forward works fine. kubectl get svc <name> -o yaml and compare with kubectl get ingress <name> -o yaml Ingress port.number equals Service spec.ports[].port (the external one). Set Ingress port.number to the Service's port, not targetPort.
The LearnKube troubleshooting order: when something's broken, debug bottom-up — first verify the Pod is Running and Ready, then verify Service endpoints exist (match B + C), then verify Ingress backend is populated (match D + E). This works because each layer depends on the one below it being correct.

One more match — for Istio users

If you're using Istio (as in your original article), there are two additional matches in the VirtualService and Gateway. The principle is the same: everything links by name or label, nothing by position.

F VirtualService gateways[] ═══ Gateway metadata.name
G Gateway selector ═══ Labels on istio-ingressgateway Pod (cluster-installed)

VirtualService http.route.destination.host uses the Service name — exactly like match D for plain Ingress. Same rule, different file.
